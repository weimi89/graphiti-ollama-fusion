# 使用 Graphiti MCP 工具的指令

## 開始任何任務之前

**始終先搜索：** 在開始工作之前，使用 `search_memory_nodes` 工具查詢相關的偏好設定和程式。

**同時搜索事實：** 使用 `search_memory_facts` 工具發現可能與您的任務相關的關係和事實資訊。

**按實體型別過濾：** 在節點搜索中指定 "Preference"（偏好）或 "Procedure"（程式）以獲得針對性結果。

**審查所有匹配項：** 仔細檢查與目前任務匹配的任何偏好、程式或事實。

## 始終儲存新的或更新的資訊

**立即捕獲需求和偏好：** 當用戶表達需求或偏好時，立即使用 `add_memory_simple` 儲存它。長文本會自動切分為多段處理。

**明確標識更新：** 如果某些內容是對現有知識的更新，請明確說明。

**清晰記錄程式：** 當您發現使用者希望如何完成某些操作時，將其記錄為程式。

**記錄事實關係：** 當您瞭解到實體之間的連線時，將這些資訊儲存為事實。

**明確分類：** 為偏好和程式標註清晰的類別，以便日後更好地檢索。

## 工作過程中

**遵循發現的偏好：** 使您的工作與找到的任何偏好保持一致。

**嚴格按照程式執行：** 如果找到適用於目前任務的程式，請嚴格按步驟執行。

**應用相關事實：** 使用事實資訊來指導您的決策和建議。

**保持一致性：** 與先前識別的偏好、程式和事實保持一致。

## 最佳實踐

**建議前先搜索：** 在提出建議之前，始終檢查是否存在既定知識。

**結合節點和事實搜索：** 對於複雜任務，同時搜索節點和事實以構建完整圖景。

**使用 center_node_uuid：** 在探索相關資訊時，圍繞特定節點進行搜索。

**優先考慮具體匹配：** 更具體的資訊優先於一般資訊。

**主動識別模式：** 如果您注意到使用者行為中的模式，考慮將其儲存為偏好或程式。

**長文本使用背景模式：** 對於較長的記憶內容，使用 `background=true` 避免阻塞等待。

**重要提醒：** 知識圖譜是您的記憶。持續使用它來提供個性化協助，尊重使用者既定的程式和事實背景。

## 可用的 MCP 工具（19 個）

### 記憶管理工具（7 個）

#### `add_memory_simple` - 添加記憶到知識圖譜

| 參數 | 類型 | 必填 | 預設值 | 說明 |
|------|------|------|--------|------|
| `name` | string | Y | - | 記憶片段的名稱 |
| `episode_body` | string | Y | - | 記憶內容（超過 800 字元自動切分） |
| `group_id` | string | | "default" | 記憶分組 ID（建議按專案隔離） |
| `source_description` | string | | "MCP Server" | 來源描述 |
| `source` | string | | "text" | 內容類型：text、json、message |
| `episode_uuid` | string | | null | 自訂 UUID（用於更新現有記憶） |
| `use_safe_mode` | boolean | | false | 安全模式（跳過實體提取，快但不可搜尋） |
| `background` | boolean | | false | 背景處理（立即返回 task_id，適合長文本） |
| `force` | boolean | | false | 跳過去重檢查（強制添加） |
| `excluded_entity_types` | array | | null | 排除的實體類型（減少不需要的提取量） |

**三種模式：**
- **完整模式（預設）**：`use_safe_mode=false`，執行完整的實體提取和關係建立，記憶可被 `search_memory_nodes` 和 `search_memory_facts` 搜尋。處理時間較長（視文本長度和模型速度而定）。啟用去重時，會先檢查是否有高度相似的既有記憶。
- **安全模式**：`use_safe_mode=true`，僅建立 EpisodicNode，秒速完成但記憶**無法**被搜尋工具找到。
- **強制模式**：`force=true`，跳過去重檢查直接添加。

**效能提示：**
- 短文本（<800 字元）：直接處理，約 30-40 秒
- 長文本（>800 字元）：自動切分為多段，每段獨立處理
- 使用 `background=true` 可避免 MCP 呼叫阻塞，透過 `get_memory_task_status` 追蹤進度
- 切分閾值和段落大小可透過環境變數 `GRAPHITI_CHUNK_THRESHOLD`、`GRAPHITI_MAX_CHUNK_SIZE` 調整

#### `add_episode_bulk` - 批量添加多筆記憶

| 參數 | 類型 | 必填 | 預設值 | 說明 |
|------|------|------|--------|------|
| `episodes` | array | Y | - | 記憶列表，每項含 `name`（名稱）和 `content`（內容） |
| `group_id` | string | | "default" | 記憶分組 ID |
| `source` | string | | "text" | 內容類型 |
| `background` | boolean | | true | 背景處理（批量通常耗時，建議保持 true） |

**使用場景：**
- 大量資料遷移（從其他系統匯入）
- 批次記錄會議紀錄、文件摘要
- 一次性導入多筆知識

#### `add_triplet` - 結構化三元組添加（跳過 LLM）

| 參數 | 類型 | 必填 | 預設值 | 說明 |
|------|------|------|--------|------|
| `source_name` | string | Y | - | 來源實體名稱（如 "Alice"） |
| `target_name` | string | Y | - | 目標實體名稱（如 "Google"） |
| `relation_name` | string | Y | - | 關係名稱（如 "works_at"） |
| `fact` | string | Y | - | 事實描述（如 "Alice works at Google"） |
| `group_id` | string | | "default" | 分組 ID |
| `source_labels` | array | | null | 來源實體標籤 |
| `target_labels` | array | | null | 目標實體標籤 |

**特點：** 直接建構節點和邊，不需要 LLM 進行實體提取，**秒速完成**。適合已知結構化知識的快速錄入。

#### `search_memory_nodes` - 搜尋記憶節點（實體）

| 參數 | 類型 | 必填 | 預設值 | 說明 |
|------|------|------|--------|------|
| `query` | string | Y | - | 搜尋關鍵字（自然語言） |
| `max_nodes` | integer | | 10 | 返回節點的最大數量 |
| `group_ids` | array | | null | 用於篩選的分組 ID 列表 |
| `entity_types` | array | | null | 實體類型過濾（如 "Preference"、"Procedure"） |
| `search_recipe` | string | | null | 搜尋策略（見 `advanced_search` 的策略表） |
| `created_after` | string | | null | 建立時間下限（ISO datetime，如 "2025-01-01T00:00:00+00:00"） |
| `created_before` | string | | null | 建立時間上限（ISO datetime） |

> **注意**：僅能搜尋以完整模式（`use_safe_mode=false`）添加的記憶。安全模式添加的記憶不會產生可搜尋的實體節點。啟用重要性追蹤時，搜尋後會自動更新存取計數。

#### `search_memory_facts` - 搜尋記憶事實（關係）

| 參數 | 類型 | 必填 | 預設值 | 說明 |
|------|------|------|--------|------|
| `query` | string | Y | - | 搜尋關鍵字 |
| `max_facts` | integer | | 10 | 返回事實的最大數量 |
| `group_ids` | array | | null | 用於篩選的分組 ID 列表 |
| `center_node_uuid` | string | | null | 圍繞特定節點搜索相關事實 |
| `edge_types` | array | | null | 關係類型過濾（如 `["works_at", "lives_in"]`） |
| `created_after` | string | | null | 建立時間下限（ISO datetime） |
| `created_before` | string | | null | 建立時間上限（ISO datetime） |
| `only_valid` | boolean | | false | 僅返回未失效的事實（排除已被新事實覆蓋的舊資訊） |

#### `advanced_search` - 進階搜尋（16 種策略）

| 參數 | 類型 | 必填 | 預設值 | 說明 |
|------|------|------|--------|------|
| `query` | string | Y | - | 搜尋關鍵字 |
| `search_recipe` | string | | "combined_rrf" | 搜尋策略 |
| `max_results` | integer | | 10 | 最大返回數量 |
| `group_ids` | array | | null | 分組過濾 |
| `center_node_uuid` | string | | null | 中心節點 UUID |

**回傳完整的 SearchResults**，包含 `nodes`（節點）、`edges`（邊）、`communities`（社群）、`episodes`（片段）四類結果。

**可用搜尋策略（search_recipe）：**

| 類別 | 策略 | 說明 |
|------|------|------|
| 綜合 | `combined_rrf` | 綜合 RRF 融合（**預設，推薦**） |
| 綜合 | `combined_mmr` | 綜合 MMR 多樣性重排 |
| 綜合 | `combined_cross_encoder` | 綜合 Cross-Encoder 精排 |
| 邊 | `edge_rrf` / `edge_mmr` / `edge_cross_encoder` | 邊搜尋（3 種排序） |
| 邊 | `edge_node_distance` / `edge_episode_mentions` | 邊搜尋（圖距離/引用次數排序） |
| 節點 | `node_rrf` / `node_mmr` / `node_cross_encoder` | 節點搜尋（3 種排序） |
| 節點 | `node_node_distance` / `node_episode_mentions` | 節點搜尋（圖距離/引用次數排序） |
| 社群 | `community_rrf` / `community_mmr` / `community_cross_encoder` | 社群搜尋 |

#### `get_episodes` - 獲取最近的記憶片段

| 參數 | 類型 | 必填 | 預設值 | 說明 |
|------|------|------|--------|------|
| `last_n` | integer | | 10 | 獲取最近記憶片段的數量 |
| `group_id` | string | | "" | 用於篩選的分組 ID |

### 知識分析工具（3 個）

#### `check_conflicts` - 檢測事實衝突

| 參數 | 類型 | 必填 | 預設值 | 說明 |
|------|------|------|--------|------|
| `source_name` | string | Y | - | 來源實體名稱 |
| `target_name` | string | Y | - | 目標實體名稱 |
| `group_id` | string | | "default" | 分組 ID |

**返回資訊：**
- `active_facts` - 目前有效的事實列表
- `invalidated_facts` - 已失效的事實列表
- `has_conflicts` - 是否存在衝突

**使用場景：** 檢查兩個實體之間是否有矛盾的資訊。例如，如果同時存在「Alice 在 Google 工作」和「Alice 在 Microsoft 工作」，而前者已被標記為失效，就可以透過此工具發現。

#### `get_node_edges` - 探索節點關係

| 參數 | 類型 | 必填 | 預設值 | 說明 |
|------|------|------|--------|------|
| `node_uuid` | string | Y | - | 節點 UUID |
| `include_inbound` | boolean | | true | 包含入邊（指向此節點的關係） |
| `include_outbound` | boolean | | true | 包含出邊（從此節點出發的關係） |
| `max_edges` | integer | | 50 | 最大返回數量 |

**返回資訊：**
- `inbound_edges` - 入邊列表（含來源節點名稱和事實描述）
- `outbound_edges` - 出邊列表（含目標節點名稱和事實描述）

#### `build_communities` - 社群檢測與聚類

| 參數 | 類型 | 必填 | 預設值 | 說明 |
|------|------|------|--------|------|
| `group_ids` | array | | null | 指定分組（留空則全部） |
| `background` | boolean | | true | 背景處理 |

**說明：** 使用 Label Propagation 演算法將相關實體自動聚類為社群，並生成社群摘要。結果可在 Web 管理介面的「社群」頁面瀏覽。

### 記憶維護工具（2 個）

#### `get_stale_memories` - 查詢過時記憶

| 參數 | 類型 | 必填 | 預設值 | 說明 |
|------|------|------|--------|------|
| `days_threshold` | integer | | 30 | 超過多少天未存取視為過時 |
| `min_access_count` | integer | | 2 | 存取次數低於此值才列入 |
| `group_id` | string | | null | 分組過濾 |
| `limit` | integer | | 50 | 最大返回數量 |

**返回資訊：** 包含 `stale_nodes` 和 `stale_edges` 列表，每項含 `uuid`、`name`/`fact`、`access_count`、`last_accessed`、`recency_score` 等資訊。

> **前提**：需要啟用 `ENABLE_IMPORTANCE_TRACKING=true`（預設已啟用），搜尋操作才會自動記錄存取計數。

#### `cleanup_stale_memories` - 清理過時記憶

| 參數 | 類型 | 必填 | 預設值 | 說明 |
|------|------|------|--------|------|
| `days_threshold` | integer | | 30 | 過時天數閾值 |
| `min_access_count` | integer | | 2 | 最低存取次數閾值 |
| `group_id` | string | | null | 分組過濾 |
| `dry_run` | boolean | | true | **預設預覽模式**（不實際刪除，僅列出將被清理的項目） |
| `limit` | integer | | 50 | 最大處理數量 |

> **安全提示：** `dry_run` 預設為 `true`，必須明確設為 `false` 才會實際刪除。清理時優先刪除邊，只有在節點變成孤立節點（無任何關係）時才會刪除節點。

### 任務管理工具

#### `get_memory_task_status` - 查詢背景任務進度

| 參數 | 類型 | 必填 | 預設值 | 說明 |
|------|------|------|--------|------|
| `task_id` | string | Y | - | 背景任務 ID（由 `add_memory_simple(background=true)` 返回） |

**返回資訊：**
- `task_id` - 任務 ID
- `status` - 狀態：pending / processing / completed / failed
- `name` - 記憶名稱
- `group_id` - 分組 ID
- `chunks_total` - 總切分段數
- `chunks_done` - 已完成段數
- `created_at` - 建立時間
- `completed_at` - 完成時間
- `result` - 處理結果（完成時）
- `error` - 錯誤訊息（失敗時）

## 工作流程範例

### 1. 開始新任務

```
1. search_memory_nodes(query="任務相關關鍵字", entity_types=["Preference", "Procedure"])
2. search_memory_facts(query="相關事實")
3. 分析搜索結果
4. 執行任務
5. add_memory_simple(name="任務結果", episode_body="詳細描述")
```

### 2. 儲存用戶偏好

```
用戶：「我喜歡用簡潔的代碼風格」
動作：add_memory_simple(
  name="代碼風格偏好",
  episode_body="用戶偏好使用簡潔的代碼風格，避免冗長的實現",
  group_id="preferences"
)
```

### 3. 記錄操作程序

```
用戶：「部署時先運行測試，然後構建，最後發佈」
動作：add_memory_simple(
  name="部署程序",
  episode_body="部署流程：1. 運行測試 2. 構建應用 3. 發佈到生產環境",
  group_id="procedures"
)
```

### 4. 圍繞特定節點探索

```
# 先找到感興趣的節點
result = search_memory_nodes(query="專案架構")
node_uuid = result[0]["uuid"]

# 圍繞該節點搜索相關事實
facts = search_memory_facts(
  query="相關資訊",
  center_node_uuid=node_uuid
)

# 探索節點的完整關係網絡
edges = get_node_edges(node_uuid=node_uuid)
```

### 5. 長文本背景處理

```
# 長文本使用背景模式，避免等待
result = add_memory_simple(
  name="會議紀錄",
  episode_body="（一份 2000 字元的會議紀錄...）",
  group_id="meetings",
  background=true
)
task_id = result["task_id"]

# 稍後檢查進度
status = get_memory_task_status(task_id=task_id)
# status.chunks_done / status.chunks_total 可知進度
```

### 6. 批量匯入知識

```
# 一次性添加多筆記憶
add_episode_bulk(
  episodes=[
    {"name": "規則1", "content": "代碼風格必須使用 4 空格縮排"},
    {"name": "規則2", "content": "所有 API 必須有錯誤處理"},
    {"name": "規則3", "content": "提交前必須通過所有測試"}
  ],
  group_id="coding-standards",
  background=true
)
```

### 7. 快速添加結構化知識

```
# 直接建立三元組，不需 LLM 提取，秒速完成
add_triplet(
  source_name="Alice",
  target_name="Google",
  relation_name="works_at",
  fact="Alice works at Google as a senior engineer",
  group_id="team"
)
```

### 8. 進階搜尋與過濾

```
# 使用特定搜尋策略
advanced_search(
  query="專案架構決策",
```

## 注意事項

### 記憶管理
- 記憶內容應該清晰、具體且有意義
- 使用適當的分組 ID 來組織不同類型的記憶（如 `preferences`、`procedures`、`decisions`）
- 定期檢查和更新過時的資訊
- 保持記憶片段的邏輯大小，避免過長或過短
- 使用描述性的名稱來便於後續檢索

### 模式選擇
- **完整模式**（預設）：記憶可被搜尋，但處理較慢（需要 LLM 進行實體提取）
- **安全模式**（`use_safe_mode=true`）：秒速完成，適合不需要搜尋的記錄
- 重要的偏好、程式和事實務必使用完整模式，以確保可被搜尋到

### 效能相關
- 長文本（>800 字元）會自動切分，無需手動分段
- 使用 `background=true` 可避免長時間阻塞
- `excluded_entity_types` 可減少不需要的實體提取，加快處理速度
- 使用 `entity_types` 參數可更精確地過濾搜索結果
- `add_triplet` 跳過 LLM，秒速完成，適合已知結構化知識
- `add_episode_bulk` 適合大量資料匯入，建議保持 `background=true`

### 搜尋策略選擇
- **一般用途**：使用預設的 `combined_rrf`（綜合 RRF 融合）
- **多樣性需求**：使用 `combined_mmr`（避免結果過於相似）
- **精確排序**：使用 `combined_cross_encoder`（精排品質最高但較慢）
- **特定類型搜尋**：使用 `edge_*`、`node_*`、`community_*` 前綴的策略
- **探索性搜尋**：使用 `node_distance` 或 `episode_mentions` 策略

### 記憶品質維護
- 定期使用 `get_stale_memories` 檢查過時記憶
- 使用 `check_conflicts` 偵測矛盾事實
- 去重功能（`add_memory_simple`）自動防止重複儲存
- 使用 `build_communities` 定期聚類，保持知識圖譜結構化

### 安全性
- 刪除操作不可逆，請先確認 UUID 正確
- `clear_graph` 清除的資料無法恢復，建議使用 `group_ids` 做局部清除
- `cleanup_stale_memories` 預設 `dry_run=true`，必須明確設為 `false` 才會刪除

## 服務端點（HTTP 模式）

使用 HTTP 傳輸模式時，可用以下端點：

| 端點 | 方法 | 說明 |
|------|------|------|
| `/mcp` | POST | MCP 協議主端點 |
| `/` | GET | Web 管理介面 |
| `/health` | GET | 健康檢查（liveness） |
| `/health/ready` | GET | 深度檢查（含 Neo4j 連線驗證） |
| `/api/stats` | GET | 儀表板統計 |
| `/api/groups` | GET | 取得所有 group_id |
| `/api/nodes` | GET | 瀏覽實體節點（分頁） |
| `/api/facts` | GET | 瀏覽事實（分頁） |
| `/api/episodes` | GET | 瀏覽記憶片段（分頁） |
| `/api/search/nodes` | GET | 向量搜尋節點 |
| `/api/search/facts` | GET | 向量搜尋事實 |
| `/api/search/advanced` | GET | 進階搜尋（16 種策略） |
| `/api/communities` | GET | 瀏覽社群節點（分頁） |
| `/api/communities/build` | POST | 觸發社群建構 |
| `/api/memory/add-bulk` | POST | 批量添加記憶 |
| `/api/memory/add-triplet` | POST | 添加三元組 |
| `/api/memory/tasks` | GET | 列出背景任務 |
| `/api/memory/tasks/{id}` | GET | 查詢單一任務 |
| `/api/analytics/stale` | GET | 查詢過時記憶 |
| `/api/analytics/cleanup` | POST | 清理過時記憶 |
| `/api/nodes/{uuid}` | DELETE | 刪除節點 |
| `/api/episodes/{uuid}` | DELETE | 刪除記憶片段 |
| `/api/facts/{uuid}` | DELETE | 刪除事實 |
| `/api/groups/{group_id}` | DELETE | 刪除整個 group |

**健康檢查範例：**
```bash
# 基本 liveness 檢查
curl http://localhost:8000/health

# 深度 readiness 檢查（含 Neo4j 連線）
curl http://localhost:8000/health/ready
```

> **注意**：SSE 模式（`--transport sse`）已不建議使用，請改用 HTTP 模式（`--transport http`）。
