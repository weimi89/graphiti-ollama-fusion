## P0 — 立即修復（影響可用性與安全性）

### 1. 硬編碼敏感資訊

- **位置**：`tools/inspect_schema.py:16`
- **問題**：硬編碼了 Neo4j 密碼 `'24927108'`
- **風險**：敏感資訊暴露在版本控制中
- **建議**：改用環境變數 `os.getenv("NEO4J_PASSWORD")`

### 2. LLM 客戶端缺乏重試與超時控制

- **位置**：`src/ollama_graphiti_client.py:163-190`
- **問題**：`_make_request()` 硬編碼 120s 超時，首次失敗直接返回空字串，無重試邏輯
- **影響**：Ollama 暫時不可用時立即失敗，無自動恢復
- **建議**：
  - 實現指數退避重試（最多 3 次）
  - 區分操作超時：嵌入 30s / LLM 生成 60s / 搜尋 30s
  - 503 狀態碼自動重試

### 3. 嵌入器請求無超時

- **位置**：`src/ollama_embedder.py:227`
- **問題**：Ollama 嵌入模型掛起時請求永遠阻塞
- **影響**：`add_memory` 操作無限期等待
- **建議**：添加 `aiohttp.ClientTimeout(total=30)`

### 4. 實體 ID 映射假設順序

- **位置**：`src/ollama_graphiti_client.py:388-389`
- **問題**：`_parse_entity_id()` 對無效 ID 使用默認值（0 或 1），多實體場景中實體 ID 衝突導致邊重複或缺失
- **影響**：知識圖譜關係準確性降低 20-40%
- **建議**：使用實體名稱而非索引進行映射，或生成 UUID

## P1 — 效能瓶頸（影響吞吐量與響應速度）

### 5. 隊列系統已定義但從未使用

- **位置**：`graphiti_mcp_server.py:77-78, 109-143`
- **問題**：定義了 `episode_queues` 和工作者，但 `_add_memory_full_mode()` 直接調用 `add_episode()`，完全繞過隊列。記憶添加阻塞主線程
- **影響**：失去非同步批處理能力，吞吐量受限
- **建議**：在 `_add_memory_full_mode()` 中將實體提取任務放入隊列，支援背景處理
- **預期收益**：吞吐量提升 3x

### 6. 搜尋無超時和分頁

- **位置**：`graphiti_mcp_server.py:481, 573`
- **問題**：搜尋操作無超時控制，大型知識圖譜上可能 >120s。`search_memory_facts()` 固定 10 個結果上限，無分頁
- **建議**：
  - 添加搜尋超時（30s）
  - 支援游標分頁
  - 根據時間複雜度動態調整 limit（搜索耗時 >5s 時自動降低）

### 7. 嵌入器批次大小固定

- **位置**：`src/ollama_embedder.py:98-126`
- **問題**：`batch_size=10` 硬編碼，不考慮文本長度差異
- **影響**：短文本（<50 字）可批量 50 個，長文本（>500 字）應降至 5 個。批量嵌入 1000 個文本可能需要 200s 而非 100s
- **建議**：根據平均文本長度動態調整批次大小
- **預期收益**：批量嵌入快 30%

### 8. Web API Cypher 查詢效能

- **位置**：`src/web_api.py:58-79, 141-155`
- **問題**：儀表板統計、節點瀏覽等 Cypher 查詢在大圖譜上可能全表掃描
- **影響**：儀表板加載需要 10-15 秒
- **建議**：
  - 確認 `group_id`, `name`, `created_at` 索引已建立並被使用
  - 使用 count 查詢優化
  - 考慮預計算統計視圖
- **預期收益**：儀表板加載 10s → 1s

### 9. test_connection 三次串行測試

- **位置**：`graphiti_mcp_server.py:717-734`
- **問題**：LLM、嵌入器、資料庫逐個串行測試
- **影響**：健康檢查需 3-10 秒
- **建議**：改用 `asyncio.gather()` 並行測試三個組件

## P2 — 穩定性與可維護性

### 10. 完整模式失敗無自動降級

- **位置**：`src/safe_memory_add.py` + `graphiti_mcp_server.py`
- **問題**：完整模式（實體提取）失敗後直接報錯，不會自動嘗試安全模式保底
- **影響**：記憶添加完全失敗，用戶需手動重試
- **建議**：
  - `_add_memory_full_mode()` 失敗時自動降級到安全模式
  - 在結果中標記 `mode_used: "safe" | "full"`
  - 添加警告標誌讓調用者知道發生了降級

### 11. Web API 刪除操作無事務性

- **位置**：`src/web_api.py:301-327`
- **問題**：刪除節點時相關邊不自動清理，可能留下孤立邊導致查詢異常
- **建議**：使用 Cypher 事務確保一致性：
  ```cypher
  MATCH (e:Episodic {uuid: $uuid})
  OPTIONAL MATCH (e)-[r]-()
  DELETE e, r
  ```

### 12. 搜尋端點無速率限制

- **位置**：`src/web_api.py:342-417`
- **問題**：`/api/search/nodes` 和 `/api/search/facts` 無速率限制，客戶端可快速發送 100+ 搜尋請求，過載 Ollama
- **影響**：單個用戶可導致服務對其他用戶不可用
- **建議**：
  - 添加速率限制中間件（10 req/min 每 IP）
  - 搜尋時超時 30s 並返回部分結果

### 13. 全域變數狀態管理

- **位置**：`graphiti_mcp_server.py:71-78`
- **問題**：`app_config`、`logger`、`graphiti_instance`、`default_group_id` 都是全域變數，無線程同步機制
- **影響**：多並發時可能發生競態條件，配置不一致
- **建議**：使用 FastMCP 的上下文或類別化管理狀態

### 14. 向量歸一化使用 print() 輸出

- **位置**：`src/ollama_embedder.py:262, 288, 302, 313`
- **問題**：對每個無效向量都使用 `print()` 輸出，大批量嵌入時頻繁 I/O 阻塞
- **影響**：大批量嵌入時日誌輸出佔總時間 10-15%
- **建議**：改用 `logging.warning()`，批量操作時只統計記錄

## P3 — 功能增強

### 前端改進

| 問題分類 | 具體問題 | 影響 |

| 搜尋功能不完整 | 向量搜尋無分頁、缺自動完成、無實時搜尋 | 中 |

## OllamaClient 細節分析

### JSON 解析復原力 (`src/ollama_graphiti_client.py:251-275`)

- 多種 JSON 提取模式（正則、代碼塊、直接解析）
- **問題**：過度複雜，不同路徑產生不同格式的結果
- **建議**：統一為單一入口的解析管線

### 空響應處理 (`src/ollama_graphiti_client.py:141-142, 517-535`)

- `_create_empty_response()` 和 `_create_minimal_response()` 功能重複
- **建議**：統一為單一 `_create_fallback_response()` 方法

### 字段映射硬編碼 (`src/ollama_graphiti_client.py:311-357`)

- 中文欄位名、舊 API 欄位名分散在代碼各處
- **建議**：建立集中式配置字典：
  ```python
  ENTITY_FIELD_MAPPING = {
      "entity_name": "name",
      "實體": "extracted_entities",
      ...
  }
  ```

### 摘要欄位遞迴修復 (`src/ollama_graphiti_client.py:481-493`)

- `_fix_summary_fields()` 遞迴遍歷整個嵌套結構，大型 JSON（>10KB）重複調用
- **影響**：大型記憶片段的實體提取加慢 30-50%
- **建議**：一次遍歷完成所有修復

## 異常處理分析 (`src/exceptions.py`)

### 自動分類邏輯 (第 326-362 行)

- `handle_exception()` 僅檢查字符串包含，容易誤分類
- 例如「embedding 已完成」被誤判為 `EmbeddingError`
- **建議**：改為基於異常類型而非消息文本

### 缺乏異常聚合

- 相同類型的異常無分組統計，無法追蹤頻率趨勢
- **建議**：添加異常計數器和定時報告

### Traceback 序列化過大 (第 86-87 行)

- 長堆疊追蹤被直接序列化到 JSON，大型回應對客戶端不友善
- **建議**：`to_dict()` 中限制 traceback 長度（1000 字符）
